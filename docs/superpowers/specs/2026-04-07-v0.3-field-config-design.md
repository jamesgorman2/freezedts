# v0.3 Field Configuration — Design Spec

Field-level configuration for `@freezed()` classes: default values and assertion validators.

## Approach

**Hybrid — types from AST, behavior from runtime.**

The parser extracts just enough from the decorator AST for type correctness (which fields have defaults). The emitter generates a constructor that reads field config from a runtime Symbol and applies defaults + assertions. Config values (functions, expressions) stay in the runtime where they're natural JavaScript.

## Runtime Changes

Replace the `WeakMap` metadata store with a `Symbol.for('freezedts:options')` static property on the class. Update `getFreezedOptions()` to read from the Symbol.

```ts
// runtime/freezed.ts
export const FREEZED_OPTIONS = Symbol.for('freezedts:options');

export function freezed(options?: FreezedOptions) {
  return function <T extends new (...args: any[]) => any>(target: T, context: ClassDecoratorContext) {
    (target as any)[FREEZED_OPTIONS] = options ?? {};
    return target;
  };
}

export function getFreezedOptions(target: new (...args: any[]) => any): FreezedOptions | undefined {
  return (target as any)[FREEZED_OPTIONS];
}
```

## Parser Changes

Two new fields extracted from the `@freezed()` decorator AST:

- **Class-level:** `hasFieldConfig: boolean` — does `@freezed({ fields: ... })` exist?
- **Property-level:** `hasDefault: boolean` — does this field's config entry have a `default` key?

The parser does NOT extract values — just detects key presence via ts-morph.

```ts
interface ParsedProperty {
  name: string;
  type: string;
  optional: boolean;
  hasDefault: boolean;  // new
}

interface ParsedFreezedClass {
  name: string;
  generatedName: string;
  properties: ParsedProperty[];
  hasFieldConfig: boolean;  // new
}
```

**AST extraction steps:**
1. Get `@freezed()` decorator's first argument (options object literal)
2. Look for `fields` property — if present, `hasFieldConfig = true`
3. For each field entry, check for `default` property — `hasDefault = true` on matching `ParsedProperty`

## Emitter Changes

### Params type

Properties with `hasDefault: true` become optional:

```ts
export type PersonParams = {
  firstName: string;
  lastName: string;
  age?: number;       // optional due to default
  email: string;
};
```

### Constructor

When `hasFieldConfig` is true, the emitter inserts a processing block before property assignment:

```ts
constructor(params: PersonParams) {
  const __options = (new.target as any)[Symbol.for('freezedts:options')];
  const __fields: Record<string, any> = __options?.fields ?? {};
  const resolved = { ...params } as Required<PersonParams>;
  for (const [key, config] of Object.entries(__fields)) {
    if ('default' in config && (resolved as any)[key] === undefined) {
      (resolved as any)[key] = config.default;
    }
    if (config.assert && !config.assert((resolved as any)[key])) {
      throw new Error(config.message ?? `Assertion failed for '${key}'`);
    }
  }
  this.firstName = resolved.firstName;
  this.lastName = resolved.lastName;
  this.age = resolved.age;
  this.email = resolved.email;
  Object.freeze(this);
}
```

When `hasFieldConfig` is false, the constructor is unchanged from v0.2.

## `with()` Interaction

The `with()` method is unchanged. It constructs a new instance via the constructor, so defaults and assertions automatically apply:

- `person.with({ age: undefined })` — default reapplied (age resolves to `0`)
- `person.with({ email: '' })` — assertion throws

## Key Behaviors

| Scenario | Result |
|----------|--------|
| Field with default, param omitted | Uses default value |
| Field with default, param provided | Uses provided value |
| `with({ field: undefined })` | Reapplies default |
| Valid value with assertion | Construction succeeds |
| Invalid value with assertion | Throws with custom message |
| Invalid value, no custom message | Throws with generic message |
| Default applied, then assertion runs | Default resolves before assertion checks |
| Class without field config | v0.2 behavior unchanged |

## Testing

### Behavior tests (`tests/field-config/`)

**Defaults:**
- Field with default, param omitted — uses default value
- Field with default, param provided — uses provided value
- Multiple fields with defaults
- `with()` on defaulted field, override omitted — preserves current value
- `with({ field: undefined })` — reapplies default

**Assertions:**
- Valid value — construction succeeds
- Invalid value — throws with custom message
- Invalid value, no custom message — throws with generic message
- Assertion on field that also has a default — default applied before assertion runs
- `with()` that violates assertion — throws

**Regression:**
- Classes without field config work as before

### Unit tests (colocated)

- `parser.test.ts` — extracts `hasFieldConfig` and `hasDefault` from AST
- `emitter.test.ts` — generates optional params type + processing block
- `freezed.test.ts` — decorator stores options via Symbol, `getFreezedOptions()` reads from Symbol

### Test fixtures (`tests/field-config/fixtures/`)

- `defaults.ts` — class with fields that have default values
- `assertions.ts` — class with fields that have assert + message
- `mixed.ts` — class with both defaults and assertions on different fields
