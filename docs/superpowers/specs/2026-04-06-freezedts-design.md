# freezedts — Design Spec

A TypeScript port of Dart's [freezed](https://github.com/rrousselGit/freezed) package for immutable class management via code generation.

## Problem

TypeScript has no native support for immutable classes with named parameters. Developers face significant boilerplate to achieve:

- Truly immutable class instances (compile-time + runtime)
- Deep copying with selective property updates
- Value equality
- Readable string representations
- Safe frozen collections

Existing immutable-data libraries target interfaces/types (e.g., Immer, Immutable.js). None provide a class-based, decorator-driven, codegen approach.

## Developer API

### Source file (`person.ts`)

```ts
import { freezed } from 'freezedts';
import { $Person } from './person.freezed.ts';

@freezed({
  equality: 'deep',
  fields: {
    age: { default: 0 },
    email: { assert: (v: string) => v.length > 0, message: 'email required' },
  }
})
class Person extends $Person {
  constructor(params: {
    firstName: string;
    lastName: string;
    age?: number;
    email: string;
  }) {
    super(params);
  }
}
```

> **Note:** TC39 stage 3 decorators do not support parameter decorators. All field-level
> configuration is placed in the `@freezed()` class decorator to keep meta-coding separate
> from the real code.

### Generated file (`person.freezed.ts`)

```ts
// person.freezed.ts — generated by freezedts, do not edit

type PersonParams = {
  firstName: string;
  lastName: string;
  age?: number;
  email: string;
};

export abstract class $Person {
  readonly firstName!: string;
  readonly lastName!: string;
  readonly age!: number;
  readonly email!: string;

  constructor(params: PersonParams) {
    // apply defaults, run assertions, assign properties, freeze
  }

  // Callable for shallow copy, proxy chain for deep copy into nested freezed types
  with(overrides: Partial<PersonParams>): Person;
  // Deep: person.with.address({ street: '123 Main' })

  equals(other: unknown): boolean;
  toString(): string;
}
```

### Conventions

| Aspect | Convention |
|--------|-----------|
| Source file | `person.ts` |
| Generated file | `person.freezed.ts` |
| Class name | `Person` |
| Generated base class | `$Person` |
| Multiple classes per file | Single `.freezed.ts` with multiple `$` classes |
| Decorators | camelCase (`@freezed`) |
| Generator invocation | `npx freezedts` or `npx freezedts --watch` |

## Features

### Immutability (v0.1)

- Generated base class assigns all constructor params as `readonly` properties
- `Object.freeze(this)` called at end of constructor — runtime immutability

### Shallow `with()` (v0.2)

Creates a new frozen instance with selective property overrides:

```ts
const p2 = person.with({ firstName: 'Jane' });
```

Returns a new instance of the concrete class, fully frozen. Original is unchanged.

### Field configuration (v0.3)

Field-level config is specified in the `@freezed()` class decorator's `fields` option:

- `default: <value>` — default value; makes the param optional
- `assert: (v) => boolean` — validation function run at construction time
- `message: string` — error message when assertion fails

```ts
@freezed({
  fields: {
    age: { default: 0 },
    email: { assert: (v: string) => v.length > 0, message: 'email required' },
  }
})
class Person extends $Person {
  constructor(params: { firstName: string; lastName: string; age?: number; email: string }) {
    super(params);
  }
}
```

### Deep `with()` (v0.4)

Chained proxy style for updating nested `@freezed` properties:

```ts
company.with.director.assistant({ name: 'John Smith' })
company.with.director({ name: 'John Doe' })
company.with({ name: 'Google', director: new Director(...) })
company.with.director({ name: 'Larry', assistant: new Assistant(...) })
```

`with` is both callable (shallow copy) and a proxy chain (deep copy into nested freezed types). Detailed implementation design deferred to start of v0.4.

### Value equality (v0.5)

`equals()` method with configurable behavior via `@freezed()`:

```ts
@freezed({ equality: 'deep' })    // default — recursive structural comparison
@freezed({ equality: 'shallow' }) // reference equality on nested non-freezed objects
```

### `toString()` (v0.6)

Readable string representation:

```ts
person.toString()
// "Person(firstName: John, lastName: Smith, age: 42)"
```

### Collection safety (v0.7)

Arrays, Maps, and Sets in properties are recursively frozen at construction time. Mutation attempts throw at runtime.

### Watch mode & CLI polish (v0.8)

- `npx freezedts --watch` — file watcher for automatic regeneration
- Clear error reporting for malformed classes, missing decorators
- Incremental regeneration (only changed files)

### Configuration & generation control (v0.9)

Per-class opt-out via `@freezed()` decorator options:

```ts
@freezed({
  copyWith: false,  // skip with() generation
  equal: false,     // skip equals() generation
})
class Person extends $Person { ... }
```

Project-wide defaults via `freezedts.config.yaml` in the project root:

```yaml
freezed:
  options:
    # Format generated .freezed.ts files (can slow down generation)
    format: true
    # Disable generation of with()/equals() for the entire project
    copyWith: false
    equal: false
```

Resolution order: per-class `@freezed()` options override `freezedts.config.yaml` defaults. If neither specifies a value, the built-in default applies (all features enabled).

### Final release (v1.0)

Integration tests across all features combined. Documentation.

## Architecture

### Two concerns, one package

- **Runtime** — the decorator and helpers that user code imports (`freezed`). Minimal, zero dependencies. Published as the main package export.
- **Generator** — the CLI/codegen tool. Depends on `ts-morph`. Consumers use it as a dev dependency only.

### Code generator pipeline

```
Source .ts files
    → ts-morph parser (extract @freezed classes, constructor params, decorators)
    → emitter (generate .freezed.ts content)
    → write .freezed.ts files
```

## Target environment

- TypeScript 6
- ESM only
- TC39 stage 3 decorators

## Delivery methodology

Each version (0.1–0.9):

1. Specify behavior tests at the beginning of the feature
2. Divide the feature into one or more tasks
3. Plan, deliver, test, refactor, and review each task
4. When all tasks complete, review and ensure all behavior tests pass (including all previous versions)

v1.0 ships after all features (0.1–0.9) are delivered and the full test suite is green.
